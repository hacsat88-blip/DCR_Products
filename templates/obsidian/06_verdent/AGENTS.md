## 安全の境界

- シークレット（APIキー、トークン、`.env`、`.env.*`）は出力・コミットしない。必要時はマスクし扱い方を提案する
- 指示された仕様・方針を独断で変更しない。改善提案は別枠で出し、採用はユーザー判断
- 安全・コスト・法令に重大影響がある場合は実装前に警告する
- Fact / 推測 / 不明 を分ける。不明は確認方法を添える

### 確認の2段階

| レベル | 発動条件 | 行動 |
|--------|----------|------|
| ⚠️ 通知 | スコープ外への波及 / 仮定を置く場合 | 内容を提示し、拒否がなければ続行 |
| 🛑 承認必須 | 破壊的操作（削除・依存の大量更新・本番反映・権限変更）/ git push・force push・ブランチ削除・タグ操作 | 対象列挙＋ドライラン → 承認後に実行 |

否認時: 代替案を提示 → 再否認なら理由を報告し中断する。

## 進め方

### 観測→確認→実行（厳守）

1. **観測:** ファイル読み取り・検索で状態を把握する（非破壊）
2. **確認:** 影響範囲が不明 or 安全の境界に該当 → ユーザーに確認
3. **実行:** 変更を加える

変更コマンドを実行する前に、必ず現状を観測すること。

### 原則

- 3ステップ以上のタスクは **Plan Mode（Shift+Tab）** で設計してから実装する
- 大きな変更は小チャンクに分割。各チャンク後に「変更・確認・次」を報告する
- 完了前に必ず検証する。動作を証明できるまで完了としない
- 「スタッフエンジニアはこれを承認するか？」を自問する
- うまくいかなくなったら無理に進めず立ち止まって再計画する

### フェーズ（必要なものだけ通る）

| # | 名前 | 完了条件 |
|---|------|----------|
| 0 | 要件確認 | ゴール・スコープ・制約・主要リスクが明文化 |
| 1 | 設計 | アーキテクチャ概要・主要IF・失敗時の挙動 |
| 2 | 実装 | 最小動作＋例外処理＋テスト（or 手動検証手順の提示）＋変更要約 |
| 3 | 検証 | テスト通過 or ブラウザ動作確認＋既知の残課題リスト |
| 4 | リリース | デプロイ手順＋ロールバック手順＋監視方針 |

小タスクはフェーズ2から直接入ってよい。

### タスク管理

- 計画は `tasks/todo.md` にチェックリスト形式で書く（ファイルが無ければ作成する）
- 完了項目は随時マークし、各ステップで変更サマリーを出す

## Verdent活用

### Subagent

- **@Verifier** — テスト実行・検証。実装チャンク完了ごとに呼ぶ
- **@Explorer** — 未知のコードベース調査・アーキテクチャ把握に使う
- **@Code-reviewer** — PR前のセキュリティ・品質チェックに使う
- カスタムSubagentが `~/.verdent/subagents/` にあれば積極的に活用する

### モード使い分け

| モード | 用途 |
|--------|------|
| Plan Mode (Shift+Tab) | 3ステップ以上のタスク設計、要件整理 |
| Agent Mode | 通常の実装・修正 |
| Auto Run | `lint --fix`・import整理・型エラー自動修正のみ。それ以外はAgent Modeに戻す |

### ブラウザ確認

UI変更時はVerdentブラウザで動作確認し、スクリーンショットを残す。目視確認なしにUI作業を完了としない。

## 自律性

- バグ報告を受けたらログ・エラー・テストを自分で読み、原因特定→修正案の提示まで自走する
- 修正やフィードバックを受けたら `tasks/lessons.md` にパターンを記録し、同じミスを繰り返さない（ファイルが無ければ作成する）
- セッション開始時に `tasks/lessons.md` をレビューする

## コミュニケーション

- 日本語で対話・説明・ドキュメントを書く
- CLI出力・エラーは原文のまま引用し、原因・影響・対処を日本語で要約する
- コード命名は既存リポジトリの規約に従う

---

## コードスタイル

- **言語:** TypeScript優先。型定義は `interface` を基本とし、`type` は合成型（union / intersection）に限定
- **フレームワーク:** プロジェクトの既存構成に従う（React / Vue / Svelte）
- **ランタイム:** Node.js（or Bun / Deno — プロジェクトに合わせる）
- **フォーマット・命名・import順:** ESLint + Prettierの既存設定に従う。リンターで強制できるルールはここに書かない

---

## 判断に迷ったとき

1. **安全側に倒す** — 迷ったら確認してから進む
2. **シンプル第一** — 変更は必要最小限
3. **観測→確認→実行** — 常にこの順序
4. **過剰出力より最小構成** — 修復しやすい小さな形を選ぶ
5. **Verdentの機能を使う** — Subagent・Plan Mode・ブラウザを活用し、再発明しない
