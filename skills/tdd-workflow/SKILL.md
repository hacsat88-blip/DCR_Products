# TDD Workflow — テスト駆動開発

すべてのコード開発で TDD 原則に従い、包括的なテストカバレッジを確保する。

## いつ使うか

- 新機能やコンポーネントの実装
- バグ修正（再現テストを先に書く）
- リファクタリング（安全ネットとして）
- API エンドポイント追加

## TDD サイクル

```
1. RED    — 失敗するテストを書く
2. GREEN  — テストを通す最小限のコードを書く
3. IMPROVE — リファクタリング（テストはグリーンのまま）
4. VERIFY — カバレッジ 80%+ を確認
```

## ステップ詳細

### Step 1: ユーザージャーニーの定義
```
As a [役割], I want to [操作], so that [利益]
```

### Step 2: テストケースの生成
各ジャーニーに対して:
- 正常系テスト
- エッジケーステスト
- エラー系テスト
- 境界値テスト

### Step 3: テスト実行（失敗確認）
テストが正しく失敗することを確認する。

### Step 4: 最小実装
テストを通すための最小限のコードを書く。

### Step 5: テスト再実行（成功確認）

### Step 6: リファクタリング
テストがグリーンのまま:
- 重複排除
- 命名改善
- パフォーマンス最適化
- 可読性向上

### Step 7: カバレッジ検証
80%+ を達成しているか確認。

## テスト種別

| 種別 | 対象 | 速度目標 |
|------|------|---------|
| Unit | 関数・ユーティリティ・コンポーネントロジック | < 50ms/テスト |
| Integration | API エンドポイント・DB 操作・サービス連携 | < 500ms/テスト |
| E2E | クリティカルなユーザーフロー全体 | < 30s/スイート |

## よくある間違い

| ❌ 間違い | ✅ 正解 |
|----------|--------|
| 実装の内部状態をテスト | ユーザーに見える振る舞いをテスト |
| CSS クラスでセレクト | セマンティックセレクタ (role, testid) |
| テスト間に依存関係 | 各テストが独立してデータを用意 |
| テストを直す | 実装を直す（テストが間違っている場合を除く） |

## ベストプラクティス

1. **テストを先に書く** — 常に TDD
2. **1テスト1アサート** — 単一の振る舞いに集中
3. **具体的なテスト名** — 何をテストしているか説明
4. **AAA パターン** — Arrange-Act-Assert
5. **外部依存をモック** — ユニットテストの分離
6. **エッジケースをテスト** — null, undefined, 空, 巨大値
7. **エラーパスをテスト** — ハッピーパスだけではない
8. **テストは高速に** — ユニットテスト < 50ms
9. **テスト後にクリーンアップ** — 副作用なし
10. **カバレッジレポートを確認** — ギャップを特定
