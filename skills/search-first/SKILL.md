# Search First — 実装前に調査する

コードを書く前に、既存の解決策を体系的に調査するワークフロー。

## いつ使うか

- 新機能の追加で既存のライブラリ/ツールがありそうなとき
- 依存関係やインテグレーションを追加するとき
- ユーティリティ・ヘルパー・抽象化を新規作成しようとしているとき

## ワークフロー

```
1. NEED ANALYSIS — 何が必要かを定義する
2. SEARCH — 並列で調査する
   - リポジトリ内 (rg/grep)
   - パッケージ (npm/PyPI/crates.io)
   - MCP サーバー
   - 既存 skills/
   - GitHub (OSS 実装)
3. EVALUATE — 候補を評価する
   - 機能適合性, メンテナンス状況, コミュニティ, ドキュメント, ライセンス, 依存数
4. DECIDE — 判断する
5. IMPLEMENT — 最小限のコードで実装する
```

## 判断マトリクス

| シグナル | アクション |
|---------|----------|
| 完全一致、メンテ良好、MIT/Apache | **Adopt** — そのまま導入 |
| 部分一致、基盤は良い | **Extend** — 導入 + 薄いラッパー |
| 弱い候補が複数 | **Compose** — 2-3個を組み合わせ |
| 見つからない | **Build** — 調査結果を踏まえて自作 |

## クイックチェック

コードを書き始める前に 4 問を自問する:

1. リポジトリ内に既存実装はあるか? → `rg` で検索
2. 一般的な問題か? → npm/PyPI を検索
3. MCP サーバーで対応可能か? → 設定を確認
4. GitHub に実装/テンプレートがあるか? → コード検索

## アンチパターン

- **即コーディング**: 既存ソリューションを確認せずに実装を開始する
- **MCP 無視**: MCP サーバーが既に機能を提供しているか確認しない
- **過剰ラッピング**: ライブラリを厚く包みすぎて利点を失う
- **依存膨張**: 小さな機能のために巨大パッケージを導入する
