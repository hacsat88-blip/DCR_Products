# Coding Standards — 全ロール共通ベースライン

> origin: ECC common/coding-style.md + security.md (DCR 向けに再構成)
> deploy.ps1: `_` プレフィクスのため deploy 対象外（参照ドキュメント）

## コードスタイル

| 項目 | 基準 |
|------|------|
| ファイル長 | 200–400 行。超えたら分割を検討 |
| 関数長 | 50 行以下。1 関数 1 責務 |
| 不変性 | デフォルトで `const` / `readonly`。変更理由がある場合のみ `let` |
| 早期リターン | ネスト 3 段以上はガード節で平坦化 |
| エラー処理 | 境界（ユーザー入力、外部 API）でのみバリデーション。内部コードは信頼する |

## 入力バリデーション（境界のみ）

- ユーザー入力: 型・範囲・長さの検証必須
- 外部 API レスポンス: スキーマ検証推奨
- 内部関数間: 過剰なチェック不要（フレームワーク保証を信頼）

## セキュリティベースライン

### プリコミットチェック（必須）

1. **秘密情報なし**: API キー、トークン、パスワードがコードに含まれていないこと
2. **SQL インジェクション対策**: パラメータ化クエリ / ORM のみ使用
3. **XSS 対策**: ユーザー入力のサニタイズ / エスケープ出力
4. **CSRF 対策**: 状態変更リクエストにトークン検証
5. **レート制限**: 公開エンドポイントにはレート制限を設定

### 秘密情報の扱い

- `.env` はリポジトリに含めない（`.gitignore` 必須）
- 環境変数または秘密管理ツール経由で注入
- ログに秘密情報を出力しない

## 品質チェックリスト

実装完了時に確認:

- [ ] 型が正しく定義されている
- [ ] エッジケースが処理されている
- [ ] エラーメッセージが有用（対処方法がわかる）
- [ ] パフォーマンスに問題がない（N+1、不要な再レンダリング等）
- [ ] テストが追加 / 更新されている
